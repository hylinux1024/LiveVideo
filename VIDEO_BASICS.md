# 视频流基础

> 解释本项目涉及的底层视频概念: 像素 / RGB / YUV / H.264 / NALU。读 [VIDEO_STREAM_FLOW.md](./VIDEO_STREAM_FLOW.md) 前建议先看这一篇。

---

## 1. 视频 = 连续图片

视频就是**快速翻动的图片**。

- 每张图叫**一帧** (frame)
- 翻动速度叫**帧率** (fps, frames per second)
- 1fps = 看幻灯片, 24fps = 电影, 30fps = 电视, 60fps = 游戏

---

## 2. 一张图片 = 像素网格

```
1920 列 × 1080 行 = 2,073,600 个像素
```

每个像素是一个**颜色**。问题: 颜色怎么用数字表示?

---

## 3. RGB: 屏幕的发光方式

LCD / OLED 屏幕每个像素由 **3 个子像素** 组成: 红、绿、蓝。

| (R, G, B) | 颜色 |
|-----------|------|
| (255, 0, 0) | 纯红 |
| (0, 255, 0) | 纯绿 |
| (0, 0, 255) | 纯蓝 |
| (255, 255, 255) | 白 |
| (0, 0, 0) | 黑 |

每个像素存 3 个字节 (R, G, B 各 0-255) → **RGB 格式**。

### RGB 的问题: 太大

| 分辨率 | 单帧 RGB | 30fps 流量 |
|--------|----------|-----------|
| 320×240 | 230 KB | 6.9 MB/s |
| 1920×1080 | 6.2 MB | 186 MB/s |
| 3840×2160 (4K) | 24.8 MB | 744 MB/s |

千兆网卡 = 125 MB/s, 家用 100M 宽带 = 12.5 MB/s。**纯 RGB 1080p 都上不了 100M 宽带**。

---

## 4. YUV: 把"亮度"和"颜色"分开存

### 生物学基础

人眼对**亮度** (明暗) 的分辨力是**色度** (颜色) 的 8 倍:

- 锥状细胞: 感知颜色 (~100 万个)
- 杆状细胞: 感知亮度 (~1.2 亿个, 20 倍)

**关键事实**: 人眼能分辨 1/256 的亮度差, 但只能分辨 1/32 的色度差。

### YUV 格式

**Y** = 亮度 (luma), **U** = 蓝色色差 (B − Y), **V** = 红色色差 (R − Y)。

```
Y: 灰度图, 完整保留 (1 像素 = 1 字节)
U: 蓝色色差, 可以缩水
V: 红色色差, 可以缩水
```

**少存点颜色, 人眼基本看不出来**。

### 三种 YUV 格式 (chroma subsampling)

记号 `4:X:Y` 描述**每 4 个亮度像素**对应多少个色度像素:

| 格式 | 比例 | 单像素字节 | 1080p 单帧 | 用途 |
|------|------|-----------|-----------|------|
| YUV444 | 4:4:4 (不缩水) | 3 | 6.2 MB | 调色 / 后期 |
| YUV422 | 4:2:2 (横向减半) | 2 | 4.1 MB | 广电 / 专业相机 |
| **YUV420** | 4:2:0 (横纵都减半) | **1.5** | **3.1 MB** | **视频编码 / 直播** |

1080p 一帧 YUV420 = `1920 × 1080 × 1.5` = **3,110,400 字节** ≈ 3.1 MB。

### YUV420 怎么存 (字节排列)

#### I420 (planar, 3 段平铺)

```
[ Y 平面: 1920×1080 = 2,073,600 字节 ]   ← 每像素 1 个 Y
[ U 平面:   960×540 =   518,400 字节 ]   ← 2×2 像素共用 1 个 U
[ V 平面:   960×540 =   518,400 字节 ]   ← 2×2 像素共用 1 个 V
                    合计 3,110,400 字节
```

#### NV12 (semi-planar, Y + UV 交错) — **Android 默认**

```
[ Y 平面:    1920×1080 = 2,073,600 字节 ]   ← 每像素 1 个 Y
[ UV 交错:   960×540×2 = 1,036,800 字节 ]   ← U₀ V₀ U₁ V₁ U₂ V₂ ...
                    合计 3,110,400 字节
```

#### 4×2 像素块的视觉化

```
  ┌──┬──┬──┬──┐
  │Y₀│Y₁│Y₂│Y₃│    ← 4 个 Y, 各 1 字节
  │Y₄│Y₅│Y₆│Y₇│
  └──┴──┴──┴──┘
        ↓ Y₀Y₁Y₂Y₃ 共用 1 个 U, 1 个 V
  ┌────┬────┐
  │ U  │ V  │       ← I420: 1+1 字节 (UV 分开存)
  └────┴────┘
  ┌────────┐
  │ U₀ V₀ │         ← NV12: 2 字节交错存
  └────────┘
```

## 5. 视频怎么压缩

YUV420 已经省了一半, 但 1080p30 仍要 **93 MB/s**, 网络扛不住。需要**再压缩 100-300 倍**。

视频有两个核心冗余可以压:

### 冗余 1: 空间冗余 (帧内)

```
天空 (纯蓝) 在一帧里, 相邻 100×100 像素颜色几乎一样
  → 不需要存 10000 个像素, 存 1 个 + "这片都是这个色" 就行
```

### 冗余 2: 时间冗余 (帧间)

```
第 1 帧: 一个人站在空地上
第 2 帧: 同一个人, 只是手稍微抬了一下
  → 第 2 帧不需要重新存整个画面, 只存"跟第 1 帧的差异"
```

## 6. H.264 的 3 种帧

| 类型 | 名字 | 含什么 | 能独立解码? | 压缩率 |
|------|------|--------|-----------|--------|
| **I 帧 (IDR)** | 关键帧 | 完整图像 | ✅ 能 | 低 (~10x) |
| **P 帧** | 预测帧 | 跟前帧的差异 | ❌ 需要前面 I/P | 中 (~30x) |
| **B 帧** | 双向预测 | 跟前/后帧的差异 | ❌ 需要前后帧 | 高 (~50x) |

时间轴:

```
时    0ms      33ms      66ms      100ms     133ms
      I        P         P         I         P
      ████     ┊         ┊         ████      ┊
      完整图    相对 I     相对 P    完整图     相对 I
                          的差异     (新 GOP)
```

**GOP** (Group Of Pictures): 两个 I 帧之间的一组帧。`ffmpeg -g 30` = 1 个 I 帧 + 29 个 P 帧。

### Profile 决定能不能用 B 帧

| Profile | 帧类型 | 延迟 | 用途 |
|---------|--------|------|------|
| **Baseline** | I + P | 最低 | **实时 / 直播** (本项目用这个) |
| Main | I + P + B | 中 | 普通视频 |
| High | I + P + B | 中 | 高清视频 |

> **为什么直播用 Baseline**: B 帧需要缓存未来帧, 引入 50-100ms 延迟, 实时场景不能要。

## 7. NALU: H.264 的"打包单位"

H.264 码流不是连续字节流, 而是被切成一段一段的 **NALU** (Network Abstraction Layer Unit)。

```
完整 H.264 码流:
[00 00 00 01] [67 ...] [00 00 00 01] [68 ...] [00 00 00 01] [65 ...] [00 00 00 01] [41 ...] ...
  起始码       SPS      起始码        PPS      起始码         IDR      起始码         P
```

### NALU 结构

```
┌──────────┬──────┬──────────────┐
│ 起始码   │ 头   │ 载荷         │
│ 00000001 │ 1字节│ N-1 字节     │
└──────────┴──────┴──────────────┘
             │
             └─ 头字节 = [F NRI Type]
                          │ │  │
                          │ │  └── 5 bit 类型
                          │ └───── 2 bit 重要性 (丢包策略)
                          └─────── 1 bit 错误位
```

### 5 种常见 NALU

| NALU type | 头字节 (F=0, NRI=11) | 含义 | 体积 |
|-----------|---------------------|------|------|
| 5 (IDR) | `0x65` = 0110 0101 | 关键帧 | 大 (几十 KB) |
| 1 (P) | `0x41` = 0100 0001 | 普通帧 | 小 (几 KB) |
| 7 (SPS) | `0x67` = 0110 0111 | 序列参数 (宽高, profile) | < 50 字节 |
| 8 (PPS) | `0x68` = 0110 1000 | 图像参数 | < 50 字节 |
| 6 (SEI) | `0x06` | 补充信息 (时间戳等) | 可变 |

> **SPS/PPS 必须在 IDR 之前到**, 解码器才能解 IDR 帧。本项目 demuxer 在 `LiveVideoEngine.cpp:202` 检测这一点。

---

## 8. 为什么解码后还是 YUV, 不是 RGB?

GPU 有专门的着色器做 **YUV → RGB 转换** (1 条 GPU 指令), 比 CPU 转换快几十倍。Android 硬件解码器默认输出 YUV420, 这是最省路径。

---

## 9. YUV 怎么在 OpenGL 里显示

Android 用 `GL_OES_EGL_image_external` 扩展, 允许 GL **直接采样** YUV 的 GraphicBuffer。Shader 大致逻辑:

```glsl
uniform samplerExternalOES uTexture;   // OES 纹理, GPU 自动识别 YUV
varying vec2 vTexCoord;

void main() {
    vec4 yuv = texture2D(uTexture, vTexCoord);
    // GPU 内部 / 着色器内部做 YUV→RGB 矩阵乘法
    gl_FragColor = colorMatrix * yuv;
}
```

这就是 `LiveVideoView.kt:64-89` 那段 OES shader 干的事。

## 11. 常见问题

### Q: ffmpeg 推流为什么要 `-pix_fmt yuv420p`?

H.264 baseline profile 规定像素格式必须是 **YUV420 planar (8-bit)**。其他格式 (YUV444, RGB) 编码出来, 低端机解不了。

### Q: H.264 编码到底做了什么?

3 件事:
1. **预测**: P 帧记录"运动矢量 + 残差", 不存像素
2. **DCT 变换**: 8×8 像素块从空间域 → 频率域, 丢掉高频
3. **熵编码**: CABAC 概率压缩

解码就是这 3 步的逆运算, 输出 YUV420 原始像素。

### Q: 一帧的延迟从哪来?

| 阶段 | 耗时 | 优化手段 |
|------|------|---------|
| 编码 (推流端) | 5-20ms | `-preset ultrafast` |
| 网络传输 | 同 WiFi < 5ms | SRT latency=120 |
| 解码 (硬解) | 5-15ms | 绑大核 + low-latency flag |
| 显示 (vsync) | 0-16ms | 30fps 下不可优化, 想更低需 60fps+ |

理论下限 ~33ms (受 vsync 限制)。
